## Objective

Ship the first Purple PHP milestone: smart functions with typed inputs, structured outputs, provider security, schema validation, policy checks, and audit logging.

## Scope

Implement the smart-function runner and an ergonomic public API for defining and running named smart functions.

Include:

* fake provider execution
* OpenAI provider implementation
* environment secret resolver
* prompt template rendering
* JSON/schema or DTO output validation
* retry and error handling
* basic allowed-provider/model policy checks
* basic budget/run policy checks
* audit records for each run
* examples for catalog/content/support workflows

### Subtask T001: Smart Function Runner

Implement the smart-function API and runner using the contracts from WP01.

### Subtask T002: Prompt and Schema Implementations

Implement prompt template rendering and structured output validation.

### Subtask T003: Environment Secrets and Basic Policy

Implement environment secret resolution, allowed-provider/model checks, and basic budget/run policy checks.

### Subtask T004: Audit Storage

Implement file-based or SQLite audit logging for smart-function runs.

### Subtask T005: OpenAI Provider

Implement the OpenAI provider request path with tests that avoid real network calls by default.

### Subtask T006: MVP Examples

Add smart-function examples for catalog/content/support workflows.

## Requirements

The MVP must not hard-code provider credentials in examples. Quickstarts should use environment secrets.

The smart-function runner must fail clearly when provider output does not satisfy the requested schema.

Audit records must not contain raw secrets.

## Acceptance Criteria

1. A developer can run a smart function with the fake provider.
2. A developer can configure OpenAI with an environment secret resolver.
3. Structured output validation succeeds for valid output and fails for invalid output.
4. Policy denial blocks execution before provider invocation.
5. Audit logging records start, completion/failure, provider/model metadata, validation status, and policy decisions.
6. Tests cover fake provider, OpenAI request construction, secret redaction, policy denial, validation failure, retry behavior, and audit logging.

## Definition of Done

* Smart functions run end to end with the fake provider.
* OpenAI provider configuration uses environment secrets.
* Schema validation failures are clear.
* Policy denial happens before provider invocation.
* Audit records redact secrets and capture provider/model/status metadata.

## Reviewer Guidance

Focus on API ergonomics, testability without external API calls, and whether MVP scope stayed smart-functions-only.

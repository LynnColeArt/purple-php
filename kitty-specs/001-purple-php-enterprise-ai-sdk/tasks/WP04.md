## Objective

Implement Purple PHP's third product layer: governed looping agents with typed tools, approvals, replayable logs, bounded execution, and runtime hooks.

## Scope

Implement an agent loop that can:

* accept a goal
* call tools repeatedly
* enforce policy before every step and tool call
* track budget, step limits, and time limits
* request approval for gated actions
* stop cleanly on completion, failure, budget exhaustion, policy boundary, or approval requirement

Implement runtime hooks for:

* before/after run
* before/after agent step
* before/after provider request/response
* before/after tool call
* before/after approval request/decision
* policy violation
* budget exceeded
* run failed

### Subtask T001: Agent Loop

Implement bounded goal-driven execution over provider and tool contracts.

### Subtask T002: Tool Execution and Side Effects

Enforce tool side-effect levels through policy.

### Subtask T003: Approvals

Implement approval request/decision contracts and execution gates.

### Subtask T004: Budgets, Limits, and Failure Recovery

Add step, time, retry, and budget controls.

### Subtask T005: Runtime Hooks

Implement typed runtime hooks around provider, tool, approval, run, step, policy, budget, and failure events.

### Subtask T006: Replayable Logs

Ensure tool and step logs can be audited and replayed where possible.

## Requirements

Hooks are extension points. Core policy checks must remain mandatory and cannot depend on hooks.

Command and file hooks should not be part of the default Composer runtime. If implemented, they should be restricted to sidecar/native/devops contexts.

Tool side-effect levels must be enforced by policy.

## Acceptance Criteria

1. A looping agent can complete a simple fake-provider workflow using read-only tools.
2. A write or destructive tool can require approval before execution.
3. Budget and step limits stop runaway loops.
4. Every step and tool call creates audit events.
5. Hooks can allow, block, warn, modify, require approval, retry, or fail a run according to typed hook results.
6. Tests prove policy cannot be bypassed by disabling or misconfiguring hooks.

## Definition of Done

* A looping agent can complete a fake-provider read-only workflow.
* Approval-gated tools stop before side effects.
* Budgets and step limits stop runaway loops.
* Hooks cannot bypass mandatory policy checks.

## Reviewer Guidance

Focus on enforcement order: policy must run in the execution path, while hooks remain extension points.

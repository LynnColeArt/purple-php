## Objective

Make Purple PHP credible for enterprise PHP estates by adding practical domain ports, integration examples, hardening direction, and deployment-mode readiness.

## Scope

Define workflow-oriented ports for CMS, ecommerce, support, and approval use cases.

Prefer action ports over broad noun repositories:

* search content
* draft content revision
* search catalog
* draft catalog update
* look up order
* classify support ticket
* request approval
* record external audit event

Add example adapters against in-memory or fixture-backed systems. These examples should demonstrate how custom CMS/ecommerce systems can integrate without making any one platform the product identity.

Prepare deployment-mode contracts for:

* pure Composer mode
* sidecar runtime mode
* native extension mode

### Subtask T001: Workflow-Oriented Domain Ports

Define ports for content, catalog, order, support, and approval workflows.

### Subtask T002: Example Domain Implementations

Add in-memory or fixture-backed examples that avoid platform-specific identity.

### Subtask T003: Tenant and Data Controls

Extend policy/audit context for tenant, user, data sensitivity, retention, and provider routing.

### Subtask T004: Deployment Readiness Contracts

Document and type pure Composer, sidecar, and native extension deployment boundaries.

### Subtask T005: Enterprise Documentation

Document the enterprise integration story without making native runtime mandatory.

## Requirements

Native and sidecar readiness must not make native installation required.

Enterprise hardening should cover tenant context, field-level sensitivity, data retention controls, provider routing, Vault/cloud-secret extension points, observability export, and SIEM/export hooks.

The implementation should avoid pretending to fully solve platform-specific adapters before the generic workflow ports are proven.

## Acceptance Criteria

1. Example domain ports demonstrate content, catalog, order, support, and approval workflows.
2. Domain examples can be used by smart functions, chat, and looping agents.
3. Policy can inspect tenant, user, data-sensitivity, and tool side-effect context.
4. Audit export shape is documented and testable.
5. Sidecar/native contracts are documented as deployment readiness, not mandatory runtime requirements.
6. The README clearly positions Purple PHP as enterprise PHP AI modernization, not a CMS-specific plugin.

## Definition of Done

* Domain ports are workflow-oriented.
* Examples cover content, catalog, order, support, and approval flows.
* Deployment readiness does not make native runtime mandatory.
* Enterprise docs explain tenant, data, audit, and provider-routing concerns.

## Reviewer Guidance

Focus on whether the domain ports are concrete enough to implement but generic enough to avoid tying Purple PHP to one CMS or ecommerce platform.

## Objective

Expand Purple PHP from smart functions into the second product layer: stateful chat and the developer CLI, while preparing tool contracts for future looping agents.

## Scope

Implement chat sessions, conversation history, streaming-ready response handling, role-aware context, and optional tool-awareness where it does not require full agent loops.

Define initial tool contracts:

* tool name and description
* input/output schemas
* side-effect level
* approval metadata
* retry metadata
* audit metadata

Add the `purple` CLI for local runs, provider checks, diagnostics, examples, and audit inspection.

### Subtask T001: Chat Session Core

Implement stateful chat sessions using existing provider, policy, secret, and audit contracts.

### Subtask T002: Streaming-Ready Response Handling

Design response handling so streaming can be supported without locking the SDK into one provider.

### Subtask T003: Tool Metadata Contracts

Implement typed PHP tool metadata without adding full looping-agent execution.

### Subtask T004: CLI Runner

Add the `purple` CLI for demos, diagnostics, provider checks, and audit inspection.

### Subtask T005: Chat and Tool Examples

Add examples for chat and tool declaration using fake providers.

## Requirements

Chat must reuse provider, secret, policy, and audit contracts from WP01 and WP02.

Tools should be represented as typed PHP capabilities, not as arbitrary shell commands.

CLI commands must be useful for local development and demos without depending on native runtime.

## Acceptance Criteria

1. A chat session can send messages through the fake provider.
2. Conversation history is persisted or passed in through a clear session interface.
3. Chat policy and audit events reuse the existing enforcement path.
4. Tool metadata can be declared and inspected without running a looping agent.
5. `purple` CLI can run at least one smart-function demo and inspect audit records.
6. Tests cover chat history, CLI behavior, tool metadata validation, and audit integration.

## Definition of Done

* Chat sessions work with the fake provider.
* CLI can run a local demo.
* Tool metadata is inspectable and validated.
* Existing policy/audit paths are reused.

## Reviewer Guidance

Focus on whether chat/tools/CLI build on the smart-function foundation instead of creating parallel runtime paths.

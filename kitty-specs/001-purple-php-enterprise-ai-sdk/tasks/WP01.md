## Objective

Create the foundation for Purple PHP as a Composer-first SDK. This work package should make the repository feel real without prematurely implementing chat, agents, hooks, sidecar, or native runtime.

## Scope

Implement the project skeleton, baseline development tooling, core namespace, test harness, fake provider, and first contract set.

Required contracts:

* `Provider`
* `ProviderRequest`
* `ProviderResponse`
* `SecretStore` or `SecretResolver`
* `PolicyEngine`
* `PolicyDecision`
* `AuditLog`
* `AuditEvent`
* `PromptTemplate`
* `SchemaValidator`
* `SmartFunction`

### Subtask T001: Composer and Tooling Baseline

Create the Composer project metadata, PSR-4 autoloading, PHP version requirement, and test/static-analysis scripts.

Files: `composer.json`, `phpunit.xml.dist`, `phpstan.neon`, `.php-cs-fixer.php`

Validation: `composer install` and configured scripts should run or fail only for missing implementation that this WP is about to add.

### Subtask T002: Core Provider Contracts

Define narrow provider request/response contracts that can support fake providers, OpenAI, and future provider adapters.

Files: `src/Contracts/**`, `tests/Contracts/**`

Validation: tests prove provider requests and responses can carry prompt/messages, model identity, metadata, and usage placeholders without binding to one provider.

### Subtask T003: Security, Policy, and Audit Contracts

Define contracts for secret resolution/redaction, policy decisions, and audit event writing.

Files: `src/Contracts/**`, `tests/Contracts/**`

Validation: tests prove secret values redact safely and policy/audit interfaces can be implemented by later packages.

### Subtask T004: Prompt, Schema, and Smart Function Contracts

Define first-pass contracts for prompt templates, schema validation, and smart function execution without implementing the full runner.

Files: `src/Contracts/**`, `tests/Contracts/**`

Validation: tests prove DTO/schema validation contracts are expressible and do not require chat, agents, hooks, CLI, sidecar, or native runtime.

### Subtask T005: Fake Provider and Test Utilities

Implement a deterministic fake provider suitable for unit tests and examples.

Files: `src/Testing/FakeProvider.php`, `tests/Testing/**`, `examples/.gitkeep`

Validation: tests prove fake provider responses are deterministic and compatible with the provider contracts.

## Requirements

The repository should include Composer metadata, autoloading, tests, static analysis or linting where practical, and examples directory scaffolding.

The fake provider must be deterministic and suitable for unit tests.

Core interfaces should be narrow and stable enough for WP02 to implement without large rewrites.

## Acceptance Criteria

1. `composer install` succeeds.
2. Test commands succeed.
3. A fake provider can return a deterministic response through the provider interface.
4. Secret values have an explicit redaction behavior.
5. Policy and audit contracts are present and covered by basic tests.
6. No public API requires chat, tool, agent, hook, sidecar, or native runtime classes.

## Definition of Done

* Composer metadata and autoloading exist.
* Core contracts are covered by tests.
* Fake provider is deterministic.
* Secret redaction behavior is explicit.
* WP01 does not implement chat, tools, agents, hooks, sidecar, or native runtime.

## Reviewer Guidance

Focus on whether the contracts are narrow enough to remain stable through WP02 and whether they avoid overfitting to OpenAI or looping agents too early.
